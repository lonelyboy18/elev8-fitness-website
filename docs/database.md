# Database

PostgreSQL 16, accessed through Prisma 7 (`@prisma/client` + `@prisma/adapter-pg` — Prisma
7 requires an explicit driver adapter; there is no bare `new PrismaClient()` reading
`DATABASE_URL` from the schema anymore). Schema source of truth: `server/prisma/schema.prisma`.

## Entity overview

```
users ──< bookings          (ON DELETE CASCADE)
users ──< payments           (ON DELETE CASCADE)
users ──< refresh_tokens     (ON DELETE CASCADE)
feedback_submissions         (standalone — no FK, public submissions)
contacts                     (standalone — no FK, public submissions)
```

Deleting a user cascades to all three owned tables in one statement — exercised directly
in `server/tests/transactions.test.ts`.

| Table                 | Notable columns                                                    |
| ---------------------- | ------------------------------------------------------------------ |
| `users`                 | `password_hash` (bcrypt), `subscription_status`, `subscription_expires` |
| `refresh_tokens`         | PK is `token_hash` (HMAC of the JWT's `jti`, never the raw token), `family_id` for rotation/reuse-detection |
| `bookings`               | `class_type`, `class_date`, `time_slot` (free text, validated by Zod not a DB enum), `status` |
| `payments`               | `razorpay_order_id`, `razorpay_payment_id` (unique once set), `status` |
| `feedback_submissions`   | `rating` (smallint 1–5)                                             |
| `contacts`               | — |

## Indexes

- `idx_booking_slot_lookup` on `bookings(class_date, time_slot, class_type, status)` —
  backs every capacity/duplicate/availability lookup, the hottest read+write path in the app.
- `uq_booking_user_slot_status` (unique) on `bookings(user_id, class_date, time_slot, status)`
  — added this phase, see below.
- Per-FK indexes on `user_id` across `bookings`, `payments`, `refresh_tokens`, plus
  `family_id` on `refresh_tokens` and `created_at`/`rating` on the two public-submission tables.

## The booking capacity race condition (fixed this phase)

**Before:** `BookingsService.create()` ran duplicate-check → capacity-count → insert as
three independent queries. Two concurrent requests for the last open slot could both pass
the capacity check before either committed the insert, over-booking the slot.

**Fix** (`server/src/modules/bookings/bookings.service.ts` +
`bookings.repository.ts`): all three steps now run inside one `withTransaction()` call
that first takes a **Postgres advisory lock**:

```sql
SELECT pg_advisory_xact_lock(hashtext('2026-08-01|05:30|bft'));
```

keyed on the exact `(date, time_slot, class_type)` triple, held for the transaction's
lifetime. This serializes every concurrent booking attempt for that one slot — and only
that slot; a request for a different date/time/class proceeds untouched. The duplicate
check, capacity count, and insert then run consistently inside that lock.

As defense-in-depth (in case the lock is ever bypassed, e.g. a direct DB write), the
schema also has a unique index `uq_booking_user_slot_status` on
`(user_id, class_date, time_slot, status)` — `status` is part of the key so a user can
still rebook a slot after cancelling it. A `P2002` violation on this index is caught and
converted to the same "You already have a booking..." conflict message.

Covered by `server/tests/bookings.test.ts`'s *"never over-books a slot when requests race
for the last spots"* test, which fires 20 concurrent booking requests at a 15-capacity
slot and asserts exactly 15 succeed.

## Migrations

Located in `server/prisma/migrations/`, applied in order. Two were hand-authored rather
than generated by `prisma migrate dev` (which needs a live database):

- `20260702180000_init` — the original schema (Phase 4).
- `20260703120000_add_booking_unique_constraint` — adds `uq_booking_user_slot_status` (this phase).

```bash
npm run prisma:migrate:deploy   # apply pending migrations (used by Docker + CI)
npm run prisma:migrate          # create+apply a new migration in local dev (needs a live DB)
npm run prisma:studio           # browse data
```

`server/Dockerfile`'s CMD runs `prisma migrate deploy` automatically before starting the
API, so containerized deployments never need a manual migration step.
